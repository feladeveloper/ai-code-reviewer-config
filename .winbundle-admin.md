# Architecture Overview

## Project Structure

```
src/
├── controllers/         # HTTP request handlers
├── services/            # Core business logic services
├── routes/              # API endpoint definitions
├── modules/             # Feature modules with isolated concerns
│   ├── identity/        # Authentication, JWT, user management, 2FA, permissions
│   ├── wallet/          # Wallet operations, balance management, transactions
│   ├── vtu/             # VTU purchase flow, product handling, bundle processing
│   ├── payment/         # Payment gateway integration, webhook handling
│   ├── draw-engine/     # Lottery/draw system, PFA, shuffling, winners
│   ├── mobile/          # Mobile-specific auth, signup, fulfillment
│   ├── admin/           # Admin operations, approvals, refunds, notifications
│   ├── inventory/       # Product/category management
│   └── notification/    # Notification queuing and delivery
├── middleware/          # Auth, permission, validation, rate limiting, error handling
├── lib/vas-vendor/      # VAS provider integrations (airtime, data, cable TV, electricity)
├── config/              # Environment, Prisma, Redis, Swagger, BullMQ
├── enums/               # Type-safe enumerations
├── types/               # TypeScript interfaces and types
├── utils/               # Logger, response formatting, error classes
└── workers/             # Background job processors (BullMQ)
```

## Design Patterns

### Middleware Chain (Request Processing Order)
1. Security (Helmet)
2. CORS & Request ID
3. Body Parsing & Compression
4. Health/Readiness Checks
5. Authentication (Auth Middleware)
6. Logging (Logging Middleware)
7. Rate Limiting (Rate Limit Middleware)
8. Validation (Validation Middleware)
9. Permission Checking (Permission Middleware)
10. Route Handler
11. Error Handling (Global Error Handler)

### Service Architecture Patterns
- **Facade Pattern**: `VTUPurchaseService` orchestrates multiple sub-services
- **Factory Pattern**: `PaymentGatewayFactory` creates appropriate payment gateway instances
- **Strategy Pattern**: Provider services implement BaseVASProvider
- **Layered Architecture**: Controllers → Services → Repositories (Prisma ORM)

## Startup Flow
1. `validateEnv()` - Validate all environment variables
2. `PrismaConnection.initialize()` - Connect to MySQL database
3. `RedisConnection.initialize()` - Connect to Redis
4. App initialization - Setup middleware and routes
5. Health/readiness endpoints available
6. Server listening on PORT
7. Graceful shutdown handling (SIGTERM, SIGINT)

## Response Format
```json
{
  "status_code": "00",
  "message": "Operation successful",
  "data": { /* payload */ },
  "code": 200
}
```

### Response Codes
- `"00"` - Success
- `"01"` - Failed
- `"02"` - Processing
- `"03"` - Insufficient Balance
- `"04"` - Invalid Payload
- `"05"` - Server Error
- `"07"` - Validation Error
- `"08"` - Not Found
- `"09"` - Unauthorized
- `"15"` - Invalid Token
- `"16"` - Missing Token

# Authentication & Identity Module

- `src/routes/identity.routes.ts` - Identity endpoints

## JWT Token Flow

### Access Token (Stateless JWT)
- Short-lived (default: 15 minutes via JWT_EXPIRES_IN)
- Contains: user ID, email, phone, permissions, refresh token ID (rtid)
- Not stored in database
- Validated purely by signature and expiration
- Format: `{ sub, email, phone, permissions, rtid, channel, iat, exp }`

### Refresh Token (Opaque)
- Long-lived (default: 30 days via REFRESH_TOKEN_EXPIRES_IN)
- Cryptographically secure random string
- Only hash stored in database (AuthToken table)
- Used to get new access tokens
- Can be revoked (logout functionality)

## Authentication Methods

### 1. Password-based Login
```
POST /api/v1/identity/login
```

### 3. Token Refresh

- `super_admin` - Full system access
- `merchant` - Business account with commission earnings
- `agent` - Sales agent with commission
- `customer` - End consumer
- `partner` - External partner with collect permissions

### Permission Examples

## Two-Factor Authentication (2FA)

### Setup Flow
1. `POST /api/v1/identity/2fa/setup` - Generate TOTP secret + QR code

### 2FA User Fields

- `twoFAMethod`: 'authenticator', 'sms', 'email'

# Wallet System

## Overview
Multi-wallet system supporting three wallet types for different transaction flows.

## Key Files
- `src/services/wallet.service.ts` - Legacy wrapper
- `src/modules/wallet/wallet.service.ts` - Core wallet logic
- `src/routes/wallet.routes.ts` - Wallet endpoints
- `src/controllers/wallet.controller.ts` - Wallet HTTP handlers

## Wallet Types

### 1. Spend Wallet (Default)
- Used for VTU purchases and general spending
- Available for customers, agents, merchants
- Can be credited (top-up) and debited (purchases)
- Primary payment source

### 2. Win Wallet (Rewards/Cashback)
- Accumulates lottery winnings and rewards
- Customers only
- Cannot be directly withdrawn
- Used first when making purchases (split debit with spend wallet)

### 3. Commission Wallet
- Earnings for agents and merchants
- Commission earnings from transactions
- Can be withdrawn to external account
- Non-customer only

## Split Debit Logic (Customers)

For customer purchases from combined balance (win + spend):
```typescript
// Check combined balance
const combined = winBalance + spendBalance

// Debit prioritizes win wallet first
if (purchaseAmount <= winBalance) {
  // Debit entire amount from win wallet
} else {
  // Debit remaining from win wallet
  const winDebit = winBalance
  // Debit remainder from spend wallet
  const spendDebit = purchaseAmount - winBalance
}
```

## Security Features
- PIN-based wallet access (4-6 digits)
- OTP requirement above threshold
- Rate limiting on transactions
- Transaction reversal tracking
- Ledger balance reconciliation
- Frozen wallet states for investigation

# Draw Engine (Lottery System)

## Overview
Provably Fair Algorithm (PFA) lottery system with Merkle tree verification, commit-reveal scheme, and cryptographic shuffling.

## Key Files
- `src/modules/draw-engine/services/draw-runner.service.ts` - Draw execution
- `src/modules/draw-engine/pfa/commitment.service.ts` - Commit-reveal scheme
- `src/modules/draw-engine/pfa/merkle.service.ts` - Merkle tree building
- `src/modules/draw-engine/services/shuffle.service.ts` - Cryptographic shuffle
- `src/modules/draw-engine/services/snapshot.service.ts` - Ticket snapshot
- `src/modules/draw-engine/services/draw-cycle.service.ts` - Draw scheduling

## Core Concepts

### Tickets
- Purchased via draw:purchase endpoint
- Purchased with wallet funds
- Tracked with batch ID (transaction reference)
- Eligible for draws based on purchase date

### Prize Levels
- Configurable number of winners per level
- Fixed amount or percentage of pot
- Ranked by prize value (1st place, 2nd place, etc.)

### Draw Cycles
- Scheduled events (weekly, monthly, custom)
- Cron-based timing
- Grace period for late entries

### PFA (Provably Fair Algorithm)
- Server generates seed hash before draw
- Client provides Merkle root
- Combined seed used for shuffling
- Server seed revealed after draw
- Anyone can verify the fairness

## Data Models

### Ticket
```typescript
{
  id: number
  ticketId: string          // Public UUID
  phoneNumber: string
  ticketLogId: number

}
```

### DrawResult
```typescript

```

## Draw Execution Flow

1. **Advisory Lock** - Acquire database lock on draw ID
2. **Snapshot Creation** - Query eligible tickets, create DrawSnapshot records
3. **Merkle Tree Building** - Hash all snapshots, build tree, get root
4. **Commitment Reveal** - Server reveals serverSeed
5. **Cryptographic Shuffle** - Fisher-Yates shuffle with combined seed
6. **Winner Allocation** - Select winners based on prize levels
7. **Result Persistence** - Update DrawResult, credit prizes
8. **Event Publishing** - Send notifications to winners

```
3. Update payment status
4. Credit wallet if funding
5. Trigger VTU if purchase
```

---

## 1. Context
Platform requires proactive monitoring of application health, external dependencies (VTU providers, payment gateways), and infrastructure components (database, Redis, queues). Need early warning system before customer-facing failures occur.

---

## 2. Decision
Implement multi-layer health check system with dedicated endpoints, background health workers, and alerting integration. Three tiers: liveness, readiness, and deep health checks.

---

## 3. Options Considered
- **Basic /health endpoint** - Simple, insufficient for complex dependencies
- **Multi-tier health checks** - Comprehensive, moderate complexity
- **External monitoring only (Datadog/New Relic)** - Powerful but costly, black-box
- **Custom health worker + Prometheus** - Full control, requires infrastructure

---

## 4. Rationale
Multi-tier approach separates concerns: liveness for container orchestration, readiness for load balancer routing, deep checks for dependency monitoring. Integrates with existing BullMQ for scheduled health jobs.

---

## 5. Consequences
**Enables:**
- Proactive failure detection
- Graceful degradation (disable failing providers)
- SLA monitoring for external dependencies
- Automated alerting before customer impact

**Limits:**
- Additional background job processing
- Must maintain health check logic per dependency
- Alert fatigue if thresholds misconfigured

---

## 6. Security and Risk Notes
- Health endpoints exposed only to internal network/load balancer
- Deep health checks require authentication
- No sensitive data in health responses
- Rate limit health endpoints (prevent DoS)
- Provider credentials tested without exposing keys
- Failed checks logged for incident analysis

---

## 7. Review Triggers
- New external dependency added
- SLA requirements change
- Monitoring stack migration
- Incident caused by undetected dependency failure

---

## 1. Context
Platform operates across multiple environments (development, test, staging, production). Each requires different configurations for databases, external APIs, feature flags, and security settings. Misconfiguration between environments has caused test data leaks and accidental production calls in development.

---

## 2. Decision
Implement strict environment isolation using validated environment variables with fail-fast startup. No environment detection by hostname or runtime inference.

---

## 3. Options Considered
- **Dotenv with validation** - Simple, explicit, fail-fast
- **Config service (Consul/etcd)** - Dynamic updates, operational overhead
- **AWS Parameter Store** - Cloud-native, adds AWS dependency
- **Hardcoded config objects** - Simple but error-prone, secrets in code

---

## 4. Rationale
Dotenv with Zod/Joi validation provides explicit configuration that fails immediately on missing/invalid values. No runtime surprises. Secrets stay out of codebase. Easy to audit and rotate credentials.

---

## 5. Consequences
**Enables:**
- Fail-fast on misconfiguration (app won't start)
- Clear separation between environments
- Easy credential rotation
- Audit trail via version control (.env.example)

**Limits:**
- Requires restart for config changes
- Must maintain .env files per environment
- Team must follow .env.example discipline

---

## 6. Security and Risk Notes
- `.env` files never committed (gitignore enforced)
- Production secrets in secure vault (not plain files)
- Different API keys per environment (test vs live)
- Database URLs isolated (no test pointing to prod)
- ALLOWED_ORIGINS strictly defined per environment
- JWT_SECRET unique per environment
- Provider sandbox mode enforced in non-production

---

## 7. Review Triggers
- Need for dynamic config updates without restart
- Multi-region deployment requirements
- Secrets rotation frequency increases
- Compliance audit requiring centralized secrets management

---

## 1. Context
Platform handles financial transactions (wallets, payments) requiring ACID compliance on writes, while reporting/analytics queries are read-heavy and can impact write performance. Need to separate concerns: transactional integrity for writes, scalable reads for reports.

---

## 2. Decision
Implement MySQL master-slave replication. Master handles all writes, slave(s) handle read-only reporting queries. Application-level read/write routing via Prisma configuration.

---

## 3. Options Considered
- **Single database** - Simple, bottleneck at scale
- **MySQL replication (master-slave)** - Proven, supported by Prisma
- **Read replicas (cloud-managed)** - AWS RDS/PlanetScale, operational simplicity, vendor lock-in
- **CQRS with separate databases** - Maximum flexibility, highest complexity

---

## 4. Rationale
MySQL replication is mature, well-documented, and Prisma supports read replicas natively. Cloud-managed replicas (AWS RDS) reduce operational burden. Keeps current stack, minimal code changes required.

---

## 5. Consequences
**Enables:**
- Write performance isolated from report queries
- Horizontal read scaling
- Report queries won't lock transaction tables
- Geographic distribution possible (future)

**Limits:**
- Replication lag (reads may be stale by seconds)
- Operational complexity for self-managed replication
- Additional database costs
- Must explicitly route queries to correct instance

---

## 6. Security and Risk Notes
- Slave has read-only credentials (cannot modify data)
- Replication uses encrypted channel (TLS)
- Reports cannot include real-time financial balances (lag risk)
- Failover requires manual or automated promotion
- Connection strings isolated per environment
- No direct slave access from public network

---

## 7. Review Triggers
- Replication lag exceeds 5 seconds consistently
- Write throughput exceeds master capacity
- Need for multi-region deployment
- Compliance requiring data residency

## 1. Context
Merchants integrate via API for VTU transactions involving money movement. Need to restrict API access to known IP addresses to prevent credential theft exploitation. Merchants should manage their own whitelist. Must not break existing merchants during rollout.

---

## 2. Decision
Implement optional IP whitelisting per merchant with self-service management via merchant portal. Enforcement at middleware level before authentication. Gradual rollout with grace period.

---

## 3. Options Considered
- **Mandatory IP whitelisting** - Maximum security, breaks existing integrations
- **Optional IP whitelisting** - Flexible, security opt-in, recommended approach
- **WAF-based IP rules** - Infrastructure level, less granular per merchant
- **API Gateway IP policies** - Vendor-specific, may not support per-user rules

---

## 4. Rationale
Optional whitelisting allows security-conscious merchants to lock down access while maintaining compatibility with existing integrations. Self-service reduces support overhead. Middleware enforcement is fast and happens before expensive auth checks.

---

## 5. Consequences
**Enables:**
- Merchants control their own security posture
- Stolen credentials useless without IP access
- Reduced attack surface for high-value accounts
- Audit trail of IP whitelist changes

**Limits:**
- Merchants with dynamic IPs may struggle
- Support requests for lockout situations
- Not all merchants will enable (security gap)

---

## 6. Security and Risk Notes
- IP stored hashed (prevent enumeration if DB leaked)
- CIDR notation support for IP ranges
- Maximum 10 IPs per merchant (prevent abuse)
- IP changes logged with timestamp and actor
- Lockout recovery via admin portal only
- IPv4 and IPv6 support required
- Bypass for internal service-to-service calls
- X-Forwarded-For header validation (prevent spoofing)

---

## 7. Review Triggers
- Credential theft incident
- Merchant requests for more than 10 IPs
- IPv6 adoption increases significantly
- Regulatory requirement for mandatory IP restriction
