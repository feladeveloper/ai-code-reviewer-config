DocCrunch Codebase Documentation

1) System Overview
Project: DocCrunch — Intelligent Document Extraction Engine
Runtime / Framework: Python (asyncio) on FastAPI
Primary LLMs: Gemini 2.5 Pro (Orchestrator Agent), Gemini 2.5 Flash (OCR Agent)
Architecture: Agentic, URL-based, 100% Server-Sent Events streaming, stateless, in-memory (tmpfs on /dev/shm — no SSD writes)
Deployment: Docker with ClamAV sidecar and a Docker-based Python code-execution sandbox sharing a tmpfs RAM disk
Key Concerns: Strict in-memory processing (no disk persistence), SSRF protection on URL fetch, malware scanning via ClamAV, XXE / XML-bomb / ZIP-bomb defences, deterministic pre-checks before LLM engagement, per-document tracing via unique request IDs, predictable token / cost reporting.

3) Product Boundaries
Goals:
- Accept URLs to any of 10 supported document formats through a single endpoint.
- Return clean, correctly ordered Markdown as the canonical output.
- Handle scanned documents and images via Vision AI orchestration.
- Process single documents and batches with equal reliability.
- Block malformed, corrupt, malicious, or unreachable documents before extraction begins.
- Stream real-time processing state via SSE; emit one result event per document.
- Operate entirely in-memory — never write documents to disk.
- Allow natural-language feedback to guide extraction when the default path is insufficient.

Non-Goals:
- DocCrunch does not manage user authentication or API key issuance.
- DocCrunch does not store, cache, or persist documents after processing.
- DocCrunch does not translate extracted content to another language.
- DocCrunch does not render documents visually or produce image output.
- DocCrunch does not accept ZIP files or any format outside the 10 supported types.
- DocCrunch does not accept multipart file uploads — documents must be accessible by URL.
- DocCrunch does not guarantee 100% extraction accuracy; the feedback mechanism is the path to correction.
- DocCrunch does not retain state between requests; every feedback call is a fresh extraction.

4) API Surface
Single endpoint:
- POST /v1/extract
  - Request body contains urls: [] (always an array, even for single-document calls) and optional feedback.
  - Response is a Server-Sent Events stream. There is no blocking JSON response alternative — every client reads the stream.
  - Emits one result event per document plus a terminal request_complete payload carrying aggregated JSON (token usage, cost, per-document status).
  - Each request is assigned a unique request_id propagated through all events and logs.

6) Architectural Invariants
- Deterministic pre-checks (format detection, validation, malware scan, SSRF / URL safety) MUST run BEFORE the orchestrator LLM is engaged. LLMs never gate safety decisions.
- No document may touch a physical SSD. The Docker code sandbox and ClamAV share a tmpfs (RAM disk) mounted at /dev/shm. Files enter RAM on fetch, pass through extractors in RAM, and exit RAM with the response.
- Service is stateless. Every feedback call is a fresh extraction with its own request_id.
- All extraction tools are invoked via the tool-based execution layer; the orchestrator dispatches, it does not implement extraction itself.
- Global Pre-Check Burst: all URLs in a batch are validated concurrently within 2 seconds of emitting request_accepted.
- The API is 100% SSE. Do NOT add a blocking JSON response path — this was explicitly removed to eliminate 504 Gateway Timeouts.

7) Security Rules (enforced in review)
- URL fetching MUST go through httpx with SSRF protection enabled. Do not bypass with raw sockets, aiohttp, or requests.
- XML parsing MUST use defusedxml. The stdlib xml module is forbidden in extractor code.
- ZIP handling MUST enforce decompression bomb and path traversal checks. Never extract to disk.
- All user-supplied URLs and feedback strings MUST be treated as untrusted input — validate before passing to any extractor or LLM prompt.
- No secrets in code. No hardcoded model keys. Every external credential is resolved from the configured secret store.
- Logs MUST NOT include raw document content, user PII, or full URLs with credentials (strip userinfo segments).
- The code sandbox MUST remain network-isolated and ephemeral. Any change that grants network access or persistent state to the sandbox requires an ADR.

8) Review Checklist (apply to every PR)
- Does the change trace to an approved Feature Spec in governance/project-docs/specs/?
- If an architectural decision is being made or changed, is there a corresponding ADR in governance/project-docs/adr/?
- Is any new dependency introduced? If yes, is it justified against the authoritative tech stack (section 5) and does it require an ADR?
- Does the change preserve in-memory-only processing? Flag any writes to disk outside /dev/shm tmpfs.
- Does the change preserve the 100% SSE contract on /v1/extract? Flag any attempt to introduce a blocking JSON response.
- Are deterministic pre-checks still upstream of the orchestrator LLM? Flag any LLM call that gates safety.
- Is SSRF protection preserved on any new URL fetch path?
- Is defusedxml used for any new XML parsing?
- Are tokens / cost still reported in the terminal request_complete event?
- Are new errors user-friendly and mapped to stable error codes?
- Is the change covered by tests? (Minimum 80% coverage per SDGP.)
- Does the change update the CHANGELOG and any relevant README / governance docs?
