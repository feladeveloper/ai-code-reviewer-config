CROM → Odoo Bridge Codebase Documentation

1) System Overview
Project: CROM → Odoo Bridge — CashToken gift/redemption ledger ingestion
Repository: cashtokenrewards/CROM-ODOO-BRIDGE
Runtime / Framework: Python 3.12 · Odoo 18.0 Enterprise addon (`crom_odoo_bridge/`)
Data stores: PostgreSQL 16 (Odoo ORM; one raw partial-unique index), Amazon S3 + SQS via boto3
Deployment: Self-hosted EC2 Odoo (systemd, apt-packaged), addons at /opt/odoo/custom_addons; AWS access via EC2 instance role `odoo-crom-bridge`
Key Concerns: THIS CODE POSTS MONEY TO A LEDGER. Idempotency, balanced double-entry, auditability, and no-PII are the four properties every change must preserve. Treat every PR as financial-impact.

3) Product Boundaries
Goals:
- Ingest CROM's daily CSV drop from s3://odoo-cashtoken/inbound/ (eu-central-1), detected event-driven via SQS within ~2 minutes, with a daily prefix sweep as reconciliation.
- Verify file integrity (SHA-256; manifest-declared when a sidecar exists, content-addressed otherwise).
- Stage every row; post one balanced account.move per gift (Dr deferred-revenue ⇄ Cr revenue, direction by Status).
- Skip duplicates quietly (counted), quarantine unparseable rows, defer posting when accounting/vendor config is missing.
- Preserve the original gift timestamp through any hold-and-release so accounting periods are never misfiled.

Non-Goals:
- No inbound HTTP endpoint — the bridge is strictly pull-based. Do NOT add controllers/webhooks for CROM.
- No S3 writes in the current permission model (results-write is a future, spec-gated feature).
- No customer PII — the feed carries none and none may be introduced (NDPR).
- No queue_job dependency — orchestration is native ir.cron by deliberate decision (Technical Plan §3).
- No direct-to-ledger SQL — all postings go through the ORM (account.move create + action_post).

4) Data Contract (CSV header row is authoritative)
Columns consumed: Reference (unique per gift → gift_ref), BatchId (per-row voucher ref → source_ref), Channel (→ reward_type), Commodity, UserPublicId (→ crom_vendor_id), UserDisplayName (→ vendor_name), AppropriationUnitCost (→ unit_value_ngn), TotalValue (→ cashtoken_quantity), TotalCost, AmountCharged (→ value_ngn — THE amount booked), Status, CreatedAt (UTC → gifted_at).
Manifest sidecar <base>.manifest.json is OPTIONAL; when present it adds declared checksum + record/value reconciliation and an environment gate.

5) Architectural Invariants (block PRs that break these without an approved ADR)
- Three-layer idempotency: unique crom_batch_log.batch_id (content-addressed: filename + SHA-256); unique crom_gift_staging.gift_ref; Postgres partial-unique index on account_move.crom_gift_ref. A file or row must never be able to post twice — any change weakening one layer needs an ADR referencing the other two.
- Per-row savepoints in staging AND posting: one bad record must never abort or poison a batch.
- Posting pre-flight: unconfigured accounting defers the batch (state 'validated'), it does not fail rows.
- Vendor mapping re-resolves at posting time (supports map-then-Reprocess for vendors mapped after staging).
- SQS messages are deleted ONLY after successful ingest (visibility-timeout retry is the failure path).
- Credentials come from the boto3 default chain (instance role). Static keys exist only as an ir.config_parameter fallback — hardcoded AWS keys anywhere are an automatic block.
- ir.cron jobs ship disabled in data XML (active eval False); enabling is an operational act, not a code default.

6) Review Checklist for PRs
- Does any change touch _create_move / _post_batch / the account codes? Verify debits == credits in all Status branches and that crom_gift_ref, crom_batch_id, crom_source are still stamped.
- Does any change touch dedup paths (_ingest_object batch identity, _populate_staging existing-refs pre-check, gift_ref constraint, account_move partial index)? Demand tests proving a re-drop and a re-sent fuller file still cannot double-post.
- Money math: value uses AmountCharged (fallback TotalCost); quantities/unit costs are floats today — flag any arithmetic that introduces rounding into booked amounts.
- Timestamps: CreatedAt parsing must remain UTC-naive-in-UTC; entry date must remain the gift date, not processing date.
- New columns from CROM: additive and optional only; a hard-required new column is a data-contract change → requirements doc bump + spec.
- Any new external call (HTTP, S3 write, new AWS service) is out of the current permission model → ADR required.
- Odoo specifics: Odoo 18 has no ir.cron.numbercall; list views use <list> not <tree>; module is installable on a DB where account module state overrides exist (custom account_extra_types coexists on target).
- Tests: crom_odoo_bridge/tests must stay green (--test-tags /crom_odoo_bridge); pipeline changes require test changes in the same PR.
- No PII fields, no logging of row-level financial data at INFO level, no secrets in code or data XML.

8) Language Conventions (authoritative for this repo)
- This is a Python / Odoo codebase. Naming follows PEP 8 and Odoo core conventions: snake_case for functions, methods, variables, and ORM field names (fields are referenced verbatim in XML views, domains, and SQL — renaming them to camelCase would break views and diverge from Odoo core). Test methods use unittest's test_* snake_case pattern.
- The global camelCase naming rule applies ONLY to JavaScript files in this repo. Do not flag snake_case in *.py files.
- Line-length guidance for Python follows the repo's existing style (~100 cols in code, wrapped prose in markdown); do not flag markdown line length.
